# wheee — Game Design

## 1. Overview

PvP тактическая игра 1v1 на сетке 7×7.
Игроки одновременно (в тиках) перемещаются и изменяют рельеф.
После серии тиков происходит катаклизм (ветер/дождь).
Цель — заставить соперника погибнуть от катаклизма.

Игра поддерживает зрительский режим с собственными механиками
и роль архитектора, управляющего катаклизмами.

---

## 2. Board

* Размер: 7×7
* Координаты: (x, y), где 0 ≤ x,y < 7
* Высота клетки: `-1 | 0 | +1`

    * `-1` — яма
    * `0` — равнина
    * `+1` — холм
* Начальное состояние: все клетки = 0

---

## 3. Role System

### 3.1 Guest (неавторизован)

* Может сразу нажать «Играть» → попадает в матч как анонимный игрок
* Может авторизоваться → становится Player
* Прогресс/статистика не сохраняются

### 3.2 Player (авторизован)

* Выбирает 1 из 3 персонажей (см. §4)
* Нажимает «Играть» → система автоматически подбирает пару (chatroulette-стиль)
* Может переключиться в режим наблюдателя
* Сохраняется статистика, рейтинг, история матчей

### 3.3 Watcher / Наблюдатель

* Переходит в режим через кнопку «Наблюдать» или по прямой ссылке на матч
* Система автоматически назначает на активный матч
* По окончании матча автоматически перекидывается на другой начавшийся матч (авто-ротация)
* Имеет собственные механики и челленджи (не влияют на ход игры игроков):
    * **Predict Winner** — предсказать победителя до начала раунда → очки
    * **Predict Move** — предсказать ход конкретного игрока в тике → очки
    * **Break Instruments** — раз за игру может «сломать» один прибор прогноза (флюгер или барометр) одному игроку на 1 раунд, лишая его подсказки

### 3.4 Architect (особая роль)

* Управляет катаклизмами матча: вручную выставляет тип погоды, направление ветра, интенсивность дождя
* Вместо рандома — ручной выбор перед каждым раундом
* Может размещать бонусные точки на карте (см. §10)
* Видит полное состояние игры (оба игрока, все высоты)

---

## 4. Characters

Авторизованный игрок выбирает одного из 3 персонажей-культур:

| Персонаж | Культура | Визуал |
|----------|----------|--------|
| 🌾 Wheat | Пшеница | Золотистое пламя / колосья |
| 🌾 Rice | Рис | Зелёное пламя / ростки |
| 🌽 Corn | Кукуруза | Оранжевое пламя / початки |

В MVP персонажи отличаются **только визуально**.
В будущем могут появиться уникальные пассивные способности.

---

## 5. Players

* Количество: 2 (A и B)
* Начальная позиция: случайная из пула симметричных пар (зеркально относительно центра 3,3):
    * `(3,5) / (3,1)` — вертикаль, расстояние 4
    * `(1,3) / (5,3)` — горизонталь, расстояние 4
    * `(2,4) / (4,2)` — диагональ, расстояние ~2.8
    * `(4,4) / (2,2)` — обратная диагональ, расстояние ~2.8
* Игрок занимает 1 клетку
* Высота игрока = высоте клетки

### 5.1 Two-Sided Map

Оба игрока играют на одной карте, но на **разных сторонах** плитки:

* Player A видит верхнюю поверхность, Player B — нижнюю
* Каждый игрок **всегда видит себя сверху**, а оппонента — на обратной стороне
* Клиент не знает, какой он игрок (A или B) — сервер отправляет уже трансформированное состояние
* Для Player B сервер **инвертирует высоты** (negateBoard) и **инвертирует действия** (raise ↔ lower)
* Результат: когда любой игрок нажимает «Raise», на его стороне появляется холм, а у оппонента — яма

---

## 6. Time System (Ticks)

* `tickDuration`: 5–7 секунд
* `ticksPerRound`: 5

Цикл:

1. Tick (игроки делают действия)
2. ...
3. Tick (5-й)
4. Weather (катаклизм)

---

## 7. Actions (1 action per tick)

Если игрок не сделал действие — оно сгорает.

### 7.1 Move

* Направления: N, S, E, W, NE, NW, SE, SW (8 направлений, как король в шахматах)
* Смещение на 1 клетку
* Нельзя выйти за карту

### 7.2 Raise

* `height += 1`
* Ограничение: максимум +1

### 7.3 Lower

* `height -= 1`
* Ограничение: минимум -1

---

## 8. Terrain Rules

* Можно изменять **любую** клетку, включая занятую игроком

---

## 9. Simultaneity

* Все действия применяются **одновременно в конце тика**
* Конфликты:

    * Raise + Lower → нет изменения
    * Move в одну клетку → оба могут стоять вместе

---

## 10. Sponsor / Challenge Bonuses

Бонусные точки на карте. Принцип: **максимально нейтральные, не влияют на баланс.**

Размещает: Architect (или система автоматически).

### 10.1 Trigger

* Бонус активируется, когда игрок **оказывается** на конкретной клетке

### 10.2 Типы бонусов

| Бонус | Эффект |
|-------|--------|
| Time Extend | +2 секунды к текущему тику (только для этого игрока) |
| Intel | Показать, что выбрал противник в текущем тике (после того как оба сделали ход) |
| Clear Sky | Убрать неопределённость прогноза на 1 раунд |

### 10.3 Ограничения

* Максимум 1 бонус на карте одновременно
* Бонус не появляется на клетке, где стоит игрок
* Не влияет на механики ветра/дождя/смерти

---

## 11. Player HUD / Forecast Instruments

Каждый игрок видит в углу экрана:

### 11.1 Wind Vane (Флюгер)

* Показывает **предполагаемое направление** будущего ветра
* Если направлений-кандидатов два — флюгер анимированно мечется между ними
* Если ветра не будет — флюгер неподвижен, приспущен

### 11.2 Rain Forecast (Барометр)

* Пиктограмма вероятности дождя:
    * ☀️ — нет дождя
    * 🌤 — маловероятен
    * ⛅ — возможен
    * 🌧 — вероятен
    * ⛈ — гарантирован

### 11.3 Breaking Instruments

* Наблюдатель (Watcher) может 1 раз за матч «сломать» один инструмент одному игроку
* Сломанный прибор показывает хаотичные данные 1 раунд, затем восстанавливается

---

## 12. Weather System

Типы:

* Wind (55%)
* Wind + Rain (45%)

**Ветер дует каждый катаклизм** — разница только в направлении. Дождь может добавляться к ветру, но не бывает дождя без ветра.

Если есть Architect — он выбирает тип и параметры.
Если Architect отсутствует — система генерирует по вероятностному распределению.

---

## 13. Wind

### 13.1 Direction

* N→S, S→N, E→W, W→E

### 13.2 Rules (для каждого игрока)

Высоты вычисляются с учётом стороны игрока (Player B видит инвертированные высоты).

1. **Проверка укрытия (upwind scan):**

    * Сканируем **против ветра** от позиции игрока до края карты
    * Если любая клетка по этому пути **выше** игрока → ветер полностью заблокирован (игрок укрыт)

2. Если не укрыт — игрок **сносится по ветру** (downwind push):

    * двигается клетка за клеткой

3. При движении:

    * если следующая клетка **выше** текущей → остановка перед ней (стена)
    * если **ниже** текущей → игрок падает в яму и останавливается (стенка ямы блокирует)
    * если равна → продолжает

4. Если достигнут край карты:

    * игрок погибает (сдувает с карты)

---

## 14. Rain

### 14.1 Basin Detection

* Низина = область клеток, окружённых клетками строго выше или краем карты
* Край карты считается стеной
* Бассейны вычисляются **отдельно для каждого игрока** с учётом его стороны карты (Player B видит инвертированные высоты)

### 14.2 Water Fill

* Вода заполняет **минимальный уровень** в каждой низине

### 14.3 Death

* Игрок погибает, если стоит на клетке с минимальным уровнем **своего** бассейна

Примеры:

* Игрок в `-1`, вокруг `0` → смерть
* Игрок в `0`, вокруг `+1` → смерть
* Игрок в `0`, но есть `-1` рядом → выживает

---

## 15. Win / Lose

Игрок проигрывает, если:

* его сдуло за карту (wind)
* он утонул (rain)

---

## 16. Core Gameplay Loop

Повторяется до победы:

1. Architect выставляет (или система генерирует) катаклизм
2. Игроки видят прогноз (Forecast Instruments)
3. 5 тиков:

    * каждый тик: 1 действие
    * Watcher может предсказывать ходы / результат
4. Катаклизм (визуализация ветра / дождя / потопа)
5. Проверка смерти
6. Если никто не погиб — следующий раунд

---

## 17. Matchmaking

* **Play**: нажал «Играть» → попал в очередь → система подбирает пару → матч начинается
* **Watch**: нажал «Наблюдать» (или перешёл по ссылке) → назначается на активный матч → авто-ротация после финиша
* **Architect**: назначается на матч вручную (через лобби) или автоматически при нехватке

---

## 18. Design Principles

* Простые правила → сложные ситуации
* Нет мгновенных гарантированных убийств
* Last-tick действия = mindgame, не exploit
* Игрок всегда имеет шанс спастись (move)
* Наблюдатель вовлечён, но не ломает баланс
* Server-authoritative: вся истина на сервере

---

## 19. MVP Scope

В MVP нет:

* высот > +1 / < -1
* особых клеток
* уникальных способностей персонажей
* мульти-юнитов

В MVP **есть**:

* 8-directional movement (как король в шахматах)
* Изменение ландшафта под игроком
* Two-sided map (серверная инверсия для Player B)
* Ветер каждый катаклизм (wind-only или wind+rain)
* Upwind shielding (укрытие за возвышенностью)
* Wind push animation (игрока сносит визуально)
* Action preview (превью raise/lower/move перед подтверждением тика)
* Guest/Player/Watcher роли
* Авторизация
* Matchmaking (автоподбор пары)
* Серверный game engine
* WebSocket real-time sync
* Forecast instruments (флюгер + барометр)
* Базовый Watcher mode (наблюдение + предсказания)

---

## 20. Tech Target

* Language: TypeScript (фронт и бэк)
* Runtime: Bun (backend + package manager + test runner)
* Framework: Vue 3 (frontend), Hono (backend HTTP)
* Render: Three.js
* Real-time: Bun native WebSocket
* Auth: OAuth2 (Google/GitHub) + JWT sessions
* DB: PostgreSQL (users, stats, match history) + Redis (matchmaking queue, live state cache)
* Monorepo: shared types between client and server

### Принцип: Server-Authoritative

* Клиент **только отправляет намерения** (action intent) и **рендерит состояние**
* Сервер — единственный источник правды
* Сервер валидирует каждое действие
* Клиент получает обновлённый GameState через WebSocket после каждого тика

## 23. Data Structures

```ts
type Height = -1 | 0 | 1

type Cell = {
  height: Height
  bonus?: BonusType | null
}

type CharacterType = 'wheat' | 'rice' | 'corn'

type Player = {
  id: 'A' | 'B'
  x: number
  y: number
  character: CharacterType
  alive: boolean
}

type Role = 'guest' | 'player' | 'watcher' | 'architect'

type ForecastData = {
  windCandidates: WindDir[]   // 0-2 направления
  rainProbability: number     // 0..1
  instrumentsBroken: {
    A: { vane: boolean; barometer: boolean }
    B: { vane: boolean; barometer: boolean }
  }
}

type BonusType = 'time_extend' | 'intel' | 'clear_sky'

type GameState = {
  board: Cell[][]
  players: Record<'A' | 'B', Player>
  tick: number
  round: number
  forecast: ForecastData
  activeBonus: { x: number; y: number; type: BonusType } | null
}

type WatcherState = {
  score: number
  predictions: WatcherPrediction[]
  breakUsed: boolean
}
```
